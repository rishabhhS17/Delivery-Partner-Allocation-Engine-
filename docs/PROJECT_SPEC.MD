# Delivery Partner Allocation Engine — Project Specification

## 1. Overview

The Delivery Partner Allocation Engine is a full-stack logistics platform that
automatically assigns the most suitable delivery rider to each incoming order,
inspired by food-delivery systems such as Swiggy and Zomato.

The core of the system is a **deterministic, rule-based allocation engine** that
scores every eligible rider against an order and selects the best one. On top of
that sits a **live simulation layer** that keeps the application populated with
realistic rider movement and order traffic at all times, and an **AI
explainability layer** that produces a plain-language reason for every decision.

The guiding design principle is a clear separation of concerns: the allocation
logic is fully predictable and reproducible, the simulation makes the system
demonstrable without real GPS hardware, and AI is used **only** to explain
decisions — never to make them.

## 2. How the System Works

### 2.1 The live map (primary screen)

The main screen is a **Live Allocation Map** of Ranchi showing roughly 20 riders
as colored markers (idle vs. busy), a set of fixed restaurant pins, and active
orders. Because a background job continuously generates orders and the simulation
moves riders, the map is never empty — the system is alive the moment it is
opened, with no real GPS feed required. The rider movement is fully simulated by
a backend loop, which stands in for the absence of real rider phones.

### 2.2 Order creation and allocation

When an order is created (either automatically by the cron job or manually by an
admin):

1. A restaurant is selected at random from the active restaurants. A customer is
   selected at random from within the restaurant's **H3 service area** (a k=5 ring
   at resolution 7, covering ~8–10 km). This prevents unrealistic cross-city pairings
   — just as Swiggy/Zomato only show you restaurants within delivery range.
2. The allocation engine **geo-filters** candidates to `ONLINE` riders inside the
   restaurant's k=2 H3 ring (~4–5 km), then ranks them by **ETAR** (Estimated Time
   to Arrive at Restaurant) combined with rating and load. Busy riders finishing a
   nearby delivery are eligible — the one who will reach the restaurant soonest wins.
3. The chosen rider transitions to a busy state, a route is drawn from the rider
   to the restaurant, and the rider marker begins moving along it.

Allocation is **deterministic**: the same inputs always produce the same
assignment, so any decision can be reproduced and debugged.

### 2.3 The allocation reason (AI layer)

Every assignment is accompanied by a human-readable explanation, for example:
*"Rahul was selected: ~1.2 km away, rated 4.5/5, no recent deliveries."*

This text may be generated by an AI model, but the **AI never selects the rider**.
The assignment is computed by the rule-based engine first; AI only phrases the
"why" afterward. A deterministic template generates the same explanation as a
fallback when AI is unavailable. This keeps the system predictable and auditable.

### 2.4 The rider state machine

Each rider progresses through a defined set of states, which drive its behavior
on the map:

| State       | Meaning                                              | Map behavior                          |
|-------------|------------------------------------------------------|---------------------------------------|
| `IDLE`      | Free and available for assignment                    | Stays at current position             |
| `ACCEPTED`  | Assigned an order, heading to the restaurant         | Moves toward the restaurant           |
| `PICKED_UP` | Collected the order, heading to the customer         | Moves toward the customer drop-off    |

On delivery the rider returns to `IDLE` and becomes available again. A separate
`availabilityStatus` (`ONLINE` / `OFFLINE`) represents whether a rider is on shift
at all — this is what a real human partner would toggle, while the simulator keeps
dummy riders permanently online. A rider is only considered for allocation when it
is both `ONLINE` and `IDLE`.

### 2.5 Movement simulation

Because there is no real GPS, rider positions are simulated with **linear
interpolation**. When an order is assigned, the system computes a route (a
straight line for simplicity, or a routing-service polyline). On every tick the
rider advances a fixed fraction (5%) of the current leg toward its destination —
so each leg completes in a predictable number of ticks. Updated positions are
saved and streamed to all connected clients over **WebSockets**, so every viewer
sees an identical, synchronized map.

The movement loop runs on the **backend**, which is the single source of truth.
If position calculations ran on the frontend, multiple viewers would drift out of
sync and nothing would persist. To conserve resources, the loop only runs while at
least one client is connected.

### 2.6 Order generation and the queue

A scheduled job creates a new order roughly every 10 minutes so the system always
has fresh activity, and an admin can also bulk-create any number of orders at once.

When demand exceeds available riders, surplus orders are **not dropped**. They are
held in a **FIFO (first-in, first-out) queue** with status `PENDING` and processed
on each tick: the moment a rider frees up, the oldest waiting order is assigned to
it. This gives the system graceful **backpressure** — under overload it slows down
rather than failing. The queue is currently database-backed; the design leaves a
clear seam to swap in a Redis-based queue (e.g. BullMQ) for higher scale.

## 3. Key Design Decisions

These decisions reflect deliberate engineering tradeoffs rather than defaults.

**H3 hexagonal geo-indexing.** The engine uses Uber's open-source `h3-js` library —
the same system Swiggy and Zomato use — to divide the city into hexagonal cells at
resolution 7 (~5 km² each). When an order arrives, the candidate pool is restricted
to `ONLINE` riders whose current H3 hex falls within a k=2 ring of the restaurant's
hex (~4–5 km radius). Hexagons tessellate perfectly with no gaps or overlaps, giving
unambiguous zone membership with O(1) lookup.

**ETAR replaces raw distance.** Distance was a proxy for "who gets there soonest."
ETAR (Estimated Time to Arrive at Restaurant = `time_to_free + travel_time`) is a
direct measure of it. A rider finishing a nearby delivery can have a lower ETAR than
an idle rider far away — and should win. The candidate pool therefore includes riders
in `ACCEPTED` and `PICKED_UP` status, not just `IDLE`.

**Weighted scoring with normalization.** ETAR, rating, and load are in different units
and cannot be added directly. Each is normalized to 0–1 first, then combined with
configurable weights (defaults: ETAR 0.50, Rating 0.30, Load 0.20).

**Load redefined as recent workload.** Load counts deliveries completed in a rolling
60-minute window, turning it into a fatigue/fairness signal that spreads work across
riders rather than overloading a few.

**Snapshot pattern for orders.** When an order is created, the restaurant's name
and coordinates are **copied onto the order itself** rather than only referenced.
If a restaurant is later removed, historical orders remain intact. This is
intentional denormalization for data integrity.

**Deterministic core, AI on the edge.** The allocation logic is rule-based and
reproducible; AI is confined to generating explanations. This preserves
predictability and debuggability.

## 4. Scoring Model

The engine runs a two-phase pipeline: **H3 geo-filter** then **weighted ETAR scoring**.

### Phase 1 — H3 Geo-filter

```
restaurantHex = h3.latLngToCell(restaurant.lat, restaurant.lng, 7)
candidateZone = h3.gridDisk(restaurantHex, 2)          // k=2 ring, ~4–5 km
```

Only `ONLINE` riders whose H3 hex is in `candidateZone` are candidates.
Riders in `IDLE`, `ACCEPTED`, and `PICKED_UP` status are all included.

### Phase 2 — Weighted Scoring

Three factors are computed per candidate, normalized to 0–1, and combined.
**Weights are not hardcoded** — runtime-configurable by the admin.

| Factor | Default weight | Score definition (normalized 0–1) |
|--------|---------------|-----------------------------------|
| ETAR | 0.50 | `1 − (ETAR / maxETAR)` — lower ETAR = higher score |
| Rating | 0.30 | `rating / 5` |
| Load | 0.20 | `1 − (recentLoad / maxLoad)` — fewer recent deliveries scores higher |

**ETAR** (Estimated Time to Arrive at Restaurant):
- IDLE riders: `ETAR = travel_time_to_restaurant`
- PICKED_UP riders: `ETAR = (1 − progress) × 120s + travel_time`
- ACCEPTED riders: `ETAR = (1 − progress) × 240s + travel_time`

`Final Score = (w_etar · ETARScore) + (w_rating · RatingScore) + (w_load · LoadScore)`,
where the weights are normalized to sum to 1 before scoring.

**Configurable weights.** The admin can adjust from the dashboard or via API.
Weighting ETAR heavily favors speed of arrival; weighting Rating heavily favors
top-performing riders. The system normalizes any supplied weights so they stay valid.

## 5. Architecture & Components

The backend logic is organized into focused modules:

| Module                | Responsibility                                                                  |
|-----------------------|---------------------------------------------------------------------------------|
| `config.js`           | Bounding box, H3 params, ETAR params, scoring weights, tick/cron timing         |
| `models.js`           | Mongoose schemas: Rider, Restaurant, Customer, Order, AllocationHistory         |
| `allocationEngine.js` | H3 geo-filter, ETAR computation, normalized weighted scoring, reason generation |
| `orderGenerator.js`   | Random order creation with H3 service area validation, single + bulk + auto-cron|
| `simulation.js`       | Tick loop: queue processing, movement, state machine, WS broadcast              |

**Data model (high level):**
- **Rider** — name, location, rating, `availabilityStatus`, movement `status`, active-order count, current order.
- **Restaurant** — name, location, active flag (admin can add / soft-delete).
- **Order** — restaurant snapshot, customer location, assigned rider, status, route + progress, allocation score and reason, lifecycle timestamps.
- **AllocationHistory** — per-decision audit trail with the full score breakdown.

**Request / event flow:**
1. Rider seeded/registered and marked online.
2. Order created (cron or admin).
3. Engine H3 geo-filters candidates and scores them by ETAR, rating, and load.
4. Best rider assigned (or order queued if none free).
5. Assignment persisted and written to allocation history.
6. Movement loop animates the rider and streams updates over WebSockets.
7. Dashboard, maps, and analytics update in real time.

## 6. Technology Stack

### Frontend
- **React.js** — application UI and component model
- **Material UI** — component library and styling
- **Axios** — HTTP client for REST calls
- **Mapbox or Google Maps** — map rendering, rider/restaurant/customer markers, routes
- **Socket.IO client** — receiving live position/state updates

### Backend
- **Node.js** + **Express.js** — REST API and server runtime
- **Socket.IO** — WebSocket transport for streaming live map state
- **node-cron** (or `setInterval`) — scheduled order generation and the movement tick
- **h3-js** — Uber's H3 hexagonal geospatial indexing library for geo-filtering

### Database
- **MongoDB** with **Mongoose** — riders, restaurants, orders, and allocation history
- *(Future scale option: **Redis** for the order queue and live rider-location caching)*

### AI Integration
- **OpenAI API** or **Gemini API** — generates the natural-language allocation
  explanation only (with a deterministic template fallback)

### Deployment
- **Frontend:** Vercel
- **Backend:** Render
- **Database:** MongoDB Atlas

---

## 7. Scope & Honest Framing

The system is best described as **a deterministic allocation engine with H3
hexagonal geo-indexing, ETAR-based predictive assignment, a live simulation
harness, and an AI explainability layer** — not as a real-time GPS dispatch
product. Travel times are Haversine approximations and rider movement is simulated.
These boundaries are intentional for a focused, demonstrable build. The architecture
directly mirrors how Swiggy and Zomato work: H3 zones define who can receive an
order, and predictive arrival time determines who actually gets it.
